Semua endpoint dilindungi JwtAuthGuard, role: ADMIN, OWNER, SUPER_ADMIN.

#### Endpoint yang harus dibuat:

**Laba Rugi**
GET /reporting/laba-rugi?from=&to=&fiscalYearId=
Logic:
- Pendapatan: sum JournalEntryLine WHERE account.type = 'REVENUE' pada periode tsb
- HPP: sum JournalEntryLine WHERE account.type = 'COGS'
- Beban Operasional: sum JournalEntryLine WHERE account.type = 'EXPENSE'
- Laba Kotor = Pendapatan - HPP
- Laba Bersih = Laba Kotor - Beban Operasional
- Return: struktur hierarki per akun, subtotal per grup, total akhir

**Neraca (Balance Sheet)**
GET /reporting/neraca?date=&fiscalYearId=
Logic:
- Aset: sum semua akun type ASSET (saldo per tanggal)
- Kewajiban: sum semua akun type LIABILITY
- Ekuitas: sum semua akun type EQUITY
- Validasi: Total Aset harus = Total Kewajiban + Ekuitas
- Return: hierarki akun dengan saldo masing-masing

**Arus Kas (Cash Flow)**
GET /reporting/arus-kas?from=&to=
Logic:
- Operasional: transaksi dari akun kas/bank yang terkait Invoice & VendorBill
- Investasi: transaksi terkait FixedAsset
- Pendanaan: transaksi modal & pinjaman
- Return: 3 seksi + saldo awal + saldo akhir

**Buku Besar**
GET /reporting/buku-besar?accountId=&from=&to=
Logic:
- Ambil semua JournalEntryLine untuk akun tertentu
- Sertakan saldo berjalan (running balance)
- Return: array transaksi kronologis dengan debit, kredit, saldo

**Trial Balance**
GET /reporting/trial-balance?from=&to=
Logic:
- Semua akun aktif dengan total debit & kredit pada periode
- Pastikan total debit = total kredit
- Return: array akun + kolom debit + kredit + selisih

**Export**
GET /reporting/export?report=laba-rugi|neraca|arus-kas|trial-balance&format=pdf|excel&from=&to=
- PDF: gunakan pdfkit atau puppeteer (cek yang sudah terinstall)
- Excel: gunakan exceljs (install jika belum ada)
- Nama file: {namaLaporan}_{from}_{to}.pdf/.xlsx

## BAGIAN 2 — PAJAK & E-FAKTUR

### 2.1 Backend: update module `tax`

Tambahkan ke apps/backend/src/modules/tax/:

**Generate e-Faktur CSV**
POST /tax/efaktur/generate
Body: { invoiceIds: string[], periode: string }
Logic:
- Ambil Invoice + InvoiceItem + Customer/Contact yang ada NPWP
- Format CSV sesuai spesifikasi DJP e-Faktur:
  - FK (Faktur Keluaran) untuk penjualan
  - Kolom: FK, KD_JENIS_TRANSAKSI, FG_PENGGANTI, NOMOR_FAKTUR, MASA_PAJAK, 
    TAHUN_PAJAK, TANGGAL_FAKTUR, NPWP_PENJUAL, NAMA_PENJUAL, NPWP_PEMBELI, 
    NAMA_PEMBELI, ALAMAT_PEMBELI, JUMLAH_DPP, JUMLAH_PPN, JUMLAH_PPNBM
- Simpan ke model EFaktur dengan status 'generated'
- Return: file CSV + summary count

**List e-Faktur**
GET /tax/efaktur?status=&periode=&page=&limit=

**Update status e-Faktur**
PATCH /tax/efaktur/:id/status
Body: { status: 'uploaded' | 'approved' | 'rejected', nomorFaktur?: string }

**Laporan Pajak Keluaran & Masukan**
GET /tax/report/keluaran?from=&to=
GET /tax/report/masukan?from=&to=
Logic:
- Keluaran: dari Invoice yang kena PPN
- Masukan: dari VendorBill yang kena PPN
- Return: daftar faktur + total DPP + total PPN

## BAGIAN 3 — REKONSILIASI BANK

### 3.1 Backend: update module `finance`

Tambahkan endpoint di apps/backend/src/modules/finance/:

**Import mutasi bank (CSV)**
POST /finance/bank/import-mutasi
Body: multipart/form-data { file: CSV, bankAccountId: string, format: 'bca'|'mandiri'|'bri'|'bni'|'general' }
Logic:
- Parse CSV sesuai format bank masing-masing
- Kolom BCA: tanggal, keterangan, cabang, amount, db/cr, saldo
- Kolom Mandiri: tanggal, keterangan, debet, kredit, saldo
- Kolom BRI/BNI: sesuaikan
- Format 'general': tanggal, keterangan, debit, kredit, saldo
- Simpan ke BankTransaction dengan status 'unmatched'
- Return: jumlah baris berhasil import

**Auto-matching transaksi**
POST /finance/bank/auto-match?bankAccountId=
Logic:
- Bandingkan BankTransaction yang 'unmatched' dengan JournalEntryLine
- Match berdasarkan: amount sama + tanggal ±3 hari
- Jika match ditemukan: update BankTransaction.status = 'matched', simpan referensi jurnal
- Return: jumlah yang berhasil di-match

**Manual match**
PATCH /finance/bank/transactions/:id/match
Body: { journalEntryLineId: string }

**Unmatch**
PATCH /finance/bank/transactions/:id/unmatch

**List mutasi bank**
GET /finance/bank/transactions?bankAccountId=&status=unmatched|matched|excluded&from=&to=

**Rekonsiliasi summary**
GET /finance/bank/reconciliation-summary?bankAccountId=&from=&to=
Return: { saldoBuku, saldoBank, selisih, unmatchedCount, matchedCount }

### 3.2 Frontend: halaman rekonsiliasi

**/keuangan/rekonsiliasi**
- Pilih rekening bank (dropdown BankAccount)
- Pilih periode
- Summary card: Saldo Buku | Saldo Bank | Selisih | Belum cocok
- Tombol "Import Mutasi Bank" → upload CSV + pilih format bank
- Tombol "Auto Match" → jalankan matching otomatis
- Tabel 2 panel:
  - Kiri: mutasi bank (tanggal, keterangan, amount, status)
  - Kanan: transaksi jurnal yang bisa di-match
- Klik baris kiri + klik baris kanan → tombol "Match Manual"
- Transaksi sudah matched tampil dengan warna hijau
- Tombol "Exclude" untuk transaksi yang memang tidak perlu di-match

## BAGIAN 4 — APPROVAL WORKFLOW

### 4.1 Backend: buat module `workflow`

Buat apps/backend/src/modules/workflow/:

Model yang dipakai: WorkflowConfig (sudah ada di schema)

**Get workflow config**
GET /workflow/config?module=purchase_order|expense|vendor_bill

**Update workflow config**
PUT /workflow/config
Body: { module, requireApproval: boolean, approverRoles: string[], minAmount?: number }

**Request approval**
POST /workflow/request
Body: { module: string, referenceId: string, referenceType: string, amount: number }
Logic:
- Cek WorkflowConfig untuk module tersebut
- Jika requireApproval = true dan amount >= minAmount:
  - Update status dokumen menjadi 'waiting_approval'
  - Kirim notifikasi ke semua user dengan approverRoles
- Jika tidak perlu approval: langsung approve otomatis

**Approve / Reject**
POST /workflow/approve
POST /workflow/reject
Body: { module, referenceId, notes?: string }
Logic:
- Update status dokumen menjadi 'approved' / 'rejected'
- Catat di AuditLog
- Kirim notifikasi ke pembuat dokumen

**List pending approvals**
GET /workflow/pending?role=
Return: semua dokumen yang menunggu approval untuk role user saat ini

### 4.2 Integrasi ke PurchaseOrder & Expense
